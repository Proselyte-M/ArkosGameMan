# 架构说明

## 总体分层

- 视图层：`qt_view.py`
  - 负责 UI 组件构建、样式应用、信号发射
  - 不直接处理业务文件读写
- 控制器层：`qt_controller.py`
  - 负责信号绑定、流程编排、状态同步
  - 将用户动作映射到服务层能力
- 服务层：`arkos_core.py`
  - 负责 ROM 文件操作、`gamelist.xml` 读写、数据校验
  - 对外提供可复用的业务接口

## 关键流程

## 1. 系统加载流程

1. 视图发出系统切换信号
2. 控制器决定模式（单系统/全库/收藏）
3. 服务层读取对应系统数据
4. 控制器完成筛选排序并回填表格

## 2. 元数据保存流程

1. 控制器根据当前选中项确定目标系统
2. 服务层在目标系统内重新加载游戏列表
3. 通过相对路径定位目标游戏并应用字段变更
4. 仅当数据有变化时写回 `gamelist.xml`

## 3. 删除流程

1. 控制器确定目标系统与目标游戏
2. 服务层在目标系统作用域处理 ROM/媒体/存档
3. 服务层按路径从目标系统列表删除并回写 XML
4. 失败时执行回滚

## 数据一致性策略

- 写操作前必须进入目标系统作用域
- 游戏定位使用相对路径匹配，而非对象引用匹配
- 收藏/全库视图仅用于展示，不能直接作为写回数据源

## 日志策略

- 入口层初始化日志格式和级别
- 控制器记录关键动作开始/完成/失败
- 服务层记录核心写盘行为和耗时
